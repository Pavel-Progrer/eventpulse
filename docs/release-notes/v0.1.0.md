# Release notes — v0.1.0

**Tag date:** 2026-05-14
**Phase:** 1 (Foundation, API, dispatch)
**Status:** Stable for portfolio review; not yet production-deployed

This is the first complete release of EventPulse. It is the deliverable for **Phase 1** of the 28-day plan, covering the domain model, HTTP API, async dispatch infrastructure, retry and dead-letter handling, structured observability, rate limiting, health probes, and a published OpenAPI 3.1 specification.

The release is a milestone, not a finish line. Phase 2 (Docker hardening, CI/CD, security scanning) and Phase 3 (semantic search, LLM-generated channel variants) follow on the same trunk.

## What's in v0.1.0

### A working, documented, tested notification dispatch service

```bash
# Submit a notification
curl -X POST http://localhost:8080/api/v1/notifications \
  -H "Authorization: Bearer $API_KEY" \
  -H "Idempotency-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{ "channel": "email", "recipient": "user@example.com",
        "payload": { "subject": "Hi", "body_text": "Hello." } }'
# → 202 Accepted with assigned id and correlation_id

# Inspect it
curl http://localhost:8080/api/v1/notifications/$ID \
  -H "Authorization: Bearer $API_KEY"
# → 200 with status, attempts[], created_at, dispatched_at
```

That round-trip covers HTTP boundary, auth, scope check, rate limiting, idempotency lookup, domain aggregate construction, event collection, transactional persistence, queue enqueue, worker pickup, channel selection, driver dispatch, attempt recording, terminal state transition, and structured logging at every step.

## Architectural highlights worth knowing

### Domain is framework-free

`src/EventPulse/Domain/` imports nothing from `Illuminate\*`. The aggregates, value objects, and domain events run under bare PHPUnit without booting Laravel. This is enforced by the test suite split: unit tests do not have the Laravel base test case in their tree.

The application layer depends on interfaces (`NotificationRepository`, `Clock`, `DomainEventDispatcher`, `NotificationDispatchQueue`); infrastructure implements them. The HTTP layer is thin: validate input → call handler → return resource. No business logic in controllers; no Eloquent in domain.

### The aggregate is the unit of consistency

`Notification` owns its attempts and its optional dead-letter mark. All state transitions go through `Notification::transitionTo()` consulting `NotificationStatus::canTransitionTo()`. Dead-lettering is the one carved-out exception — it bypasses the enum check because it carries a domain invariant the enum cannot know: dead-lettering requires at least one failed attempt to exist.

Domain events are *collected*, not dispatched inline. `Notification::request()` raises `NotificationRequested` into a `$pendingEvents` array; the application layer pulls them after persistence with `pullPendingEvents()` and hands them to the `DomainEventDispatcher`. This makes the order well-defined and makes testing the aggregate trivial.

### Replay produces a new aggregate

A replayed dead-lettered notification is not a state transition on the original. It is a fresh aggregate, linked back via `replay_of_id`, with its own correlation id and attempt history. The original notification is preserved intact for audit. The `replay` operation is idempotent — a second call with the same `Idempotency-Key` returns the original reply; a second call with a different key on an already-replayed notification returns `409 ALREADY_REPLAYED`.

### Idempotency is a real contract, not a Redis cache

The dedup key is `(api_key_id, idempotency_key)` indexed unique. First writer wins. Duplicate same-body → `200` with the original response. Duplicate different-body → `409 IDEMPOTENCY_CONFLICT`. Same key value across different API keys does not collide. The trade-off (DB hit per request vs Redis cache) is documented in `SubmitNotificationHandler`'s docblock; the choice is revisitable if the dedup rate ever dominates the endpoint's latency budget.

### Webhook signing is replay-resistant

Outbound webhooks carry an HMAC-SHA256 signature over `{timestamp}.{json_body}` plus the timestamp header. Receivers verify by recomputing the HMAC with their shared secret and rejecting timestamps outside a 5-minute window. A reference verification snippet ships in the README and in the OpenAPI spec, in plain Python so it's implementable in any language.

### Channels are pluggable, dispatcher is dumb

`ChannelDispatcher` does not know about email, webhooks, or SMS. It receives a `Channel` enum and looks up the registered `ChannelDriver`. Adding a new channel — Slack, Discord, Microsoft Teams — is a new driver registration; the dispatcher does not change. The SMS driver is a stub today because the freelance use case is webhooks and email; the contract is real, replacement is a single class.

## Known limitations (deliberate)

These are not bugs. They are documented Phase 1 scope boundaries.

- **No production Dockerfile.** A `docker-compose.yaml` ships for local development. The production multi-stage distroless build is a Phase 2 deliverable.
- **CI runs tests + Pint only.** Psalm, PHPStan, Trivy, gitleaks, and Roave Security Advisories ship in Phase 2 with their baselines.
- **SMS is a stub.** Real provider integration (e.g., Twilio, MessageBird) is a single-driver-class change. The interface is real.
- **Inbound webhook verification is documentation, not code.** ADR-0008 documents how a receiver verifies an EventPulse-signed webhook. There are no inbound callback endpoints in Phase 1 because the API surface does not need them.
- **No semantic search, no LLM variants.** Phase 3.
- **No frontend.** Permanent. See [ADR-0001](./adr/0001-scope-and-exclusions.md).
